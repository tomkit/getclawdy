## Overview

Clawdy is a macOS menu bar companion app. Lives entirely in the macOS status bar (no dock icon, no main window). Clicking the menu bar icon opens a custom floating panel with companion voice controls. Uses push-to-talk (ctrl+option) to capture voice input, transcribes it **on-device with Apple Speech**, and sends the transcript + screenshots of the user's screen(s) to the user's **own installed coding-assistant CLI** (`claude` or `codex`). The model responds with text, which is spoken aloud via **local `AVSpeechSynthesizer` TTS**. A blue cursor overlay can fly to and point at UI elements the model references on any connected monitor.

**Clawdy is free and fully local.** There is no Cloudflare Worker, no API keys, and no metering. Responses are billed to the user's existing Claude Code / Codex subscription via the CLI they're already signed into. Nothing sensitive ships in the app.

> Clawdy uses the local-CLI + local-voice architecture below. Earlier cloud paths (a paid Cloudflare Worker proxy, ElevenLabs TTS, AssemblyAI STT, and OpenAI) were removed.

## Architecture

- **App Type**: Menu bar-only (`LSUIElement=true`), no dock icon or main window
- **Framework**: SwiftUI (macOS native) with AppKit bridging for menu bar panel and cursor overlay
- **Pattern**: MVVM with `@StateObject` / `@Published` state management
- **AI Chat**: The `CoachEngine` protocol, implemented by two CLI-backed engines that shell out (Process) to the user's installed CLI and bill their subscription:
  - `ClaudeCodeEngine` → a WARM, long-lived `claude` process (`ClaudePersistentSession`) in print mode with `--input-format stream-json`. Screenshots are sent INLINE as base64 image blocks on stdin (no Read tool, `num_turns=1`). The process is spawned at app launch and kept warm for the ENTIRE app lifetime (no idle teardown; an unexpected death self-heals via a capped respawn; `shutdown()` terminates it on engine switch / app quit) so every push-to-talk pays ~no cold start and Claude remembers the conversation server-side. See "Latency architecture" below.
  - `CodexEngine` → `codex` CLI in non-interactive mode (`codex exec --json -i <image> -`). Unchanged — still one-shot per request.
- **Engine selection**: Detected at startup AND re-detected on menu-panel open (`CoachEngineRegistry.rescan()` — a CLI installed while Clawdy runs becomes selectable without a relaunch); only installed engines are selectable in the panel (the picker gates + the setter re-guards); choice persisted to UserDefaults and re-validated against the live set. Defaults to Claude Code when both are present; a simple "No coding assistant found — install one" prompt (with copyable install commands) when neither is.
- **Autonomous Research mode (Claude-only, Slice 1)**: A SEPARATE, long-running subsystem alongside the quick-answer path — it does NOT touch the warm `ClaudePersistentSession` and uses no Claude native subagents. The warm voice agent is the ROUTER: its system prompt tells it to either answer quickly itself (unchanged) or, for big "research the web and build me a page" asks, emit a single `[RESEARCH] <task>` directive as its entire reply. `ResearchDirective` parses that directive (TTS is suppressed for it via `looksLikeResearchPrefix`), and `CompanionManager` hands the task to `ResearchSessionManager`. The coordinator runs a DEDICATED `claude -p` process (via the one-shot `CLIProcessRunner`, fully isolated — its own process/Task/state) in two phases: PLAN (`--permission-mode plan`; `ResearchPlanAnalyzer` decides "needs clarifying input vs proceed") → EXECUTE (`--resume <id> --permission-mode acceptEdits` with a narrow `--allowedTools WebSearch WebFetch Write` allowlist, `--max-budget-usd` cap). The session id is **pre-minted up front** (a lowercased UUID) and passed as `--session-id <uuid>` on the PLAN phase (claude echoes it back verbatim — empirically verified against 2.1.198); EXECUTE continues it via `--resume <uuid>`, and both phases append to the one `~/.claude/projects/<sanitized-cwd>/<uuid>.jsonl` transcript. BOTH phases run with their **CWD set to a durable STABLE per-session dir** `~/Library/Application Support/Clawdy/research/<sessionId>/` (never `$HOME`; `--add-dir` also grants it, HOME stays only in the env for auth) — this is required, not just for the file sandbox: Claude Code persists each session under a project keyed by the working directory, so `--resume <id>` only resolves when the execute phase runs in the SAME directory the plan phase created the session in. (Running the plan phase in `$HOME` while execute ran elsewhere made `--resume` fail with "No conversation found with session ID" and every real research run end in `phaseFailed`.) The dir is stable+durable (not the old throwaway `/var/folders/.../clawdy-research/<uuid>`) so the finished session stays RESUMABLE for the planned continue-conversation + History features. Every run is indexed in a lightweight Clawdy-owned JSON manifest (`ResearchManifestStore` → `~/Library/Application Support/Clawdy/research/manifest.json`): a `kind: research` entry written on start (status `running`) and updated on completion/failure/stop (status + `updatedAt` + `deliverablePath`). The warm quick-answer `ClaudePersistentSession`'s OWN `session_id` is also captured READ-ONLY from its stream init and recorded as a `kind: root` manifest entry — with NO change to the warm session's working directory, args, or behavior (fully de-commingling the root transcript by moving its CWD is out of scope). ALL execute constraints — run the tools now, ONE self-contained page (inline `<style>` only, no CDN/JS), the ABSOLUTE `report.html` path — are folded into the `-p` user message (`composeExecuteUserMessage`), the channel guaranteed to survive `--resume` (we do not depend on `--append-system-prompt` being applied to a resumed session). Neither phase uses `--bare`, so both bill the user's subscription. A single rotating status line (mapped by `ResearchStreamParser` → `ResearchStatusLine`) drives the dedicated global `ResearchStackedOverlayController` (a compact `.statusBar`, `sharingType=.none` panel with an explicit fixed frame — never the quick-answer cursor overlay, which stays fully separate); clicking the overlay when the plan needs input opens the `sharingType=.none` clarifying-question panel (`ResearchClarificationPanelManager`); the deliverable HTML opens in an embedded `WKWebView` window (`ResearchResultsWindowController`), never the default browser. A visible Stop control cancels the run (SIGTERM to the research process only — never the warm session). **Engine-by-kind selection**: research is driven behind the `ResearchEngine` protocol by the engine matching the user's SELECTED coach engine (falling back to whichever CLI is installed) — `CompanionManager.resolveResearchEngineSelection` resolves `(kind, binaryPath)`, `ResearchSessionManager` builds `ClaudeResearchEngine` for `.claudeCode` and `CodexResearchEngine` for `.codex`, and each engine owns its own per-session output-dir + transcript strategy (promoted to the protocol). **Codex research parity (Stage 3, v1)**: a Codex user's `[RESEARCH]` now routes to a WORKING single-turn Codex path (`CodexResearchEngine`) — NO plan/clarify phase (`runPlanPhase` returns `.readyToExecute` immediately; `supportsPlanPhase=false`), a client-minted run-id dir, one `codex exec -s workspace-write -C <dir> --add-dir <dir> -c tools.web_search=true --json -` turn (all constraints folded into the stdin prompt — Codex has no system-prompt flag), the `thread_id` captured POST-HOC from `thread.started` as the `--resume` handle (`supportsPreMintedSessionID=false`), spend bounded by the execute timeout only (no `--max-budget-usd`), and a follow-up via `codex exec resume <thread_id>`. The manifest run is tagged `engineKind: codex`, and its transcript path is backfilled once the thread id is known so History transcript works. **Codex parity (Stages A/C/D)**: the Codex `thread_id` is now PERSISTED at `thread.started` ingestion to `ResearchManifestStore` (`codexThreadId`, with a fallback that recovers it from the stored `rollout-<ts>-<thread_id>.jsonl` transcript path), so a killed/timed-out run still keeps a resume handle. A finished Codex run is therefore RECONSTRUCTABLE + follow-up-able via `codex exec resume <thread_id>` (gated on an available thread id — a run that captured none stays non-followable), and History's 'Resume in Terminal' launches the producing engine's native resume (`codex resume <thread_id>` for Codex, `claude --resume <id>` for Claude). Codex plans inline in its single `codex exec` turn (no native plan/build split to switch on), so we PASS THROUGH rather than emulate a plan/clarify phase. The Claude research path is 100% unchanged.
- **Speech-to-Text**: Apple Speech framework, on-device (`AppleSpeechTranscriptionProvider`). No cloud STT.
- **Text-to-Speech**: Pluggable via the `SpeechTTSProviding` protocol. Default + automatic fallback is local `AVSpeechSynthesizer` (`LocalSpeechTTSClient`). Optionally, the user can bring their OWN ElevenLabs API key (`ElevenLabsTTSClient`) which calls the ElevenLabs API DIRECTLY (no proxy/worker). Key is stored in the macOS Keychain (`TTSKeychainStore`). If ElevenLabs is selected but the key is missing/invalid or the request fails/times out/rate-limits, the utterance silently falls back to Apple TTS (`TTSProviderSelection`).
- **Screen Capture**: ScreenCaptureKit (macOS 14.2+), multi-monitor, on-demand per hotkey (≤800px longest-edge JPEG q0.5 per display — a round-2 latency reduction from 1280px/q0.8 that the investigation proved the model reads identically; named constants in `CompanionScreenCaptureUtility`). ALL displays are still encoded for full multi-monitor context. For Claude the JPEG is sent inline as a base64 image block; for Codex it's written to a private temp dir and passed by path.
- **Annotation Overlay (graffiti drawing)**: While the user holds the Control+Option push-to-talk chord, the cursor overlay becomes drawable — left-click-drag paints Clawdy-blue strokes on the display under the cursor, so the user can circle/point at UI while talking. Because the overlay is `sharingType = .none` (invisible to ScreenCaptureKit), the strokes are composited into the captured screenshot in software (`AnnotationImageCompositor`) BEFORE it is sent, so the model literally sees the annotations (base image drawn upright first, then the stroke layer under a flipped CTM). A snapshot of the drawn strokes is taken synchronously at PTT release (into `pendingAnnotationStrokes`) so a delayed/fallback transcript can't drop them, then the strokes fade out. Because Control+Option is also a common window-manager chord (e.g. Ctrl+Option+Arrow to move a window), teardown is hardened against a missed modifier key-up: `GlobalPushToTalkShortcutMonitor` reconciles against live hardware modifier flags when the CGEvent tap re-enables and synthesizes the missed release; there is an `NSApplication.willResignActive` backstop; pressing **Escape** aborts annotation mode (a true cancel-without-submit, with a “press esc to exit” affordance hint); and `AnnotationModeWatchdog` force-recovers a wedge — it is a wedge DETECTOR that re-checks live flags on each interval, so it can never interrupt a legitimate long hold.
- **Voice Input**: Push-to-talk via `AVAudioEngine`. System-wide keyboard shortcut via listen-only CGEvent tap.
- **Element Pointing**: The model embeds `[POINT:x,y:label:screenN]` tags in responses (instructed via the system prompt passed to each CLI). `CompanionManager.parsePointingCoordinates` parses these into an ORDERED `[ParsedPoint]` (each carrying its character offset AND its `spokenPosition` in the tag-stripped text); the overlay walks the targets in order, mapping coordinates to the correct monitor and animating the blue cursor along a bezier arc. **Audio-synced pointing (Stage 4, ElevenLabs only)**: when ElevenLabs TTS is active and returns character-level timestamps, the cursor advance to each target is SYNCED to (and slightly LEADS) the spoken audio — `CompanionManager` maps each POINT's `spokenPosition` into the spoken clip's `alignment` array to get the audio time the element is named, then schedules `advanceToNextPointingTarget` against THAT clip's `AVAudioPlayer` playhead at `wordStart − leadSeconds`. The lead + anchor strategy are named tunables in `PointAudioSyncTuning` (default lead ~0.4s, anchor = start of the word before the tag); the offset→word→time math is the pure, unit-tested `PointAudioSyncMapper`. Two traps are flagged in-code: (1) use the `alignment` object, NOT `normalized_alignment` (number/abbrev expansion shifts indices and desyncs); (2) each clip's alignment times are relative to that clip's OWN zero, so per-clip re-basing maps each point to the word within the SAME clip. Gracefully degrades to the fixed per-point dwell (Stage 1–3) whenever timing is unavailable (Apple TTS, ElevenLabs failed/empty alignment) — driven by `CompanionManager.pointingAdvanceIsAudioSynced`.
- **Concurrency**: `@MainActor` isolation, async/await throughout
- **Analytics**: PostHog via `ClawdyAnalytics.swift` (email collection + usage; unrelated to the LLM backend)

### CLI engine invocations

**Claude Code** (`ClaudeCodeEngine` → `ClaudePersistentSession`):
```
claude -p \
  --append-system-prompt "<coaching system prompt>" \
  --tools "" \
  --input-format stream-json \
  --output-format stream-json --verbose --include-partial-messages \
  --exclude-dynamic-system-prompt-sections
```
The process is launched ONCE and kept warm. Each push-to-talk turn is written to its **stdin** as one NDJSON `user` message: a text block (image labels + the spoken prompt) followed by one base64 `image` block per screenshot (`ClaudeStreamJSONMessage`). The model sees the screens directly — no Read tool, no temp dir, `num_turns=1`. Parses `text_delta` events for sentence-by-sentence TTS and the final `result` event for the authoritative answer. Does NOT use `--bare` (that would force API-key auth). The `[POINT:...]` protocol still works inline. **`--safe-mode` is gated on the `useClaudeCustomizations` setting ("Use my Claude Code setup"), which defaults ON and controls BOTH the warm quick-answer path AND the research plan+execute paths together.** Default (ON) = `--safe-mode` OMITTED, so the user's own `claude` customizations (CLAUDE.md, skills, plugins, hooks, MCP servers) load — a spoken answer (and a research run) reflects the same configured environment the user gets in their terminal. This is the intended product default: bringing your own `claude` means bringing your own setup. Turning the setting OFF adds `--safe-mode` (isolation). COMPATIBILITY CAVEAT for the OFF path: as of `claude` 2.1.198, `--safe-mode` (or `CLAUDE_CODE_SAFE_MODE=1`, or `--setting-sources ""`) combined with `--input-format stream-json` makes the CLI exit 0 with EMPTY stdout — the warm process then hits EOF with no `result` and the turn throws `processEndedUnexpectedly`. The warm path guards this specific empty-output signature (`ClaudePersistentSessionPolicy.isLikelySafeModeEmptyOutput` → `SessionError.isolationModeUnsupported`) and speaks specific guidance to turn the setting back on, instead of the generic "hit a snag" toast. (2.1.199 fixed the upstream bug, but the guard stays since the CLI auto-updates.) The research path does not use `--input-format stream-json`, so it is not exposed to that empty-output combo. The app-lifetime prewarm hides any cold-start cost either way.

> **Latency architecture** (measured against `claude` 2.1.195, release → first audio): the original Read-tool path was `num_turns=2`, ~9s wall, and TTS waited for the whole reply. Inline images drop it to `num_turns=1`, ~3–3.5s wall; the warm process removes ~1.5s cold start on 2nd+ turns; streaming TTS speaks the first sentence at first text (~2.6–3.1s) instead of after the full result; and the screenshot capture is hoisted to push-to-talk PRESS so it overlaps the user speaking. Net: ~8.5–9s → ~2.6–3.5s. The full investigation is in the `claude-engine-latency-findings` memory.

**Codex** (`CodexEngine`):
```
codex exec --skip-git-repo-check -s read-only -C "<temp dir>" --json \
  -i "<temp dir>/screen1.jpg" [-i ...] -
```
Codex still writes screenshots to a private per-request temp dir and grants read-only access to just that dir. Prompt (with the coaching system prompt folded in — `codex exec` has no system-prompt flag) is fed on **stdin** via the trailing `-`. Images attach natively with `-i`. Final answer is the `agent_message` item in the `--json` stream. The Claude latency work above was NOT applied to Codex. **App-lifetime persistent session does NOT apply to Codex**: `codex exec` is a non-interactive one-shot — it reads its prompt from stdin and exits when the turn ends, so there is no long-lived interactive stdin session to hold open the way `claude --input-format stream-json` allows. Codex therefore keeps its existing per-request spawn (`CoachEngine.shutdown()` is a no-op for it). This is deliberate, not faked — the engine selection / detection is unchanged.

Binaries are resolved via PATH plus common install dirs (`~/.local/bin`, `/opt/homebrew/bin`, `/usr/local/bin`, every `~/.nvm/versions/node/*/bin`) because a Finder-launched app has a minimal PATH (`CLIBinaryResolver`).

### Key Architecture Decisions

**Menu Bar Panel Pattern**: The companion panel uses `NSStatusItem` for the menu bar icon and a custom borderless `NSPanel` for the floating control panel. This gives full control over appearance (dark, rounded corners, custom shadow) and avoids the standard macOS menu/popover chrome. The panel is non-activating so it doesn't steal focus. A global event monitor auto-dismisses it on outside clicks.

**Cursor Overlay**: A full-screen transparent `NSPanel` hosts the blue cursor companion. It's non-activating, joins all Spaces, and never steals focus. The cursor position, response text, waveform, and pointing animations all render in this overlay via SwiftUI through `NSHostingView`.

**Global Push-To-Talk Shortcut**: Background push-to-talk uses a listen-only `CGEvent` tap instead of an AppKit global monitor so modifier-based shortcuts like `ctrl + option` are detected more reliably while the app is running in the background.

**Shared URLSession for AssemblyAI**: A single long-lived `URLSession` is shared across all AssemblyAI streaming sessions (owned by the provider, not the session). Creating and invalidating a URLSession per session corrupts the OS connection pool and causes "Socket is not connected" errors after a few rapid reconnections.

**Transient Cursor Mode**: When "Show Clawdy" is off, pressing the hotkey fades in the cursor overlay for the duration of the interaction (recording → response → TTS → optional pointing), then fades it out automatically after 1 second of inactivity.

## Build & Run

```bash
# Open in Xcode
open Clawdy.xcodeproj

# Select the Clawdy scheme, set signing team, Cmd+R to build and run

# Known non-blocking warnings: Swift 6 concurrency warnings,
# deprecated onChange warning in OverlayWindow.swift. Do NOT attempt to fix these.
```

**Prefer building from Xcode for day-to-day work** — running `xcodebuild` from the terminal can invalidate TCC (Transparency, Consent, and Control) permissions, so the app may need to re-request screen recording, accessibility, etc. after a CLI build.

For a headless build/test gate (CI), use:

```bash
xcodebuild -project Clawdy.xcodeproj -scheme Clawdy -destination 'platform=macOS' build CODE_SIGNING_ALLOWED=NO
xcodebuild test  -project Clawdy.xcodeproj -scheme Clawdy -destination 'platform=macOS' -only-testing:ClawdyTests CODE_SIGNING_ALLOWED=NO
```

A shared scheme (`Clawdy.xcscheme`) includes the test target so `xcodebuild test` works.

## Runtime requirements (no backend)

Clawdy has **no backend**. It needs at least one coding-assistant CLI installed and signed in:

- Claude Code: `npm install -g @anthropic-ai/claude-code`, then run `claude` once to sign in.
- Codex: `npm install -g @openai/codex`, then `codex login`.

There are no API keys, no Cloudflare Worker, and no `worker/` directory. The product name is **Clawdy** (`PRODUCT_NAME`); the internal target, scheme, source directory, and Swift module are all named `Clawdy` too (`PRODUCT_MODULE_NAME` = `Clawdy`).

## Code Style & Conventions

### Variable and Method Naming

IMPORTANT: Follow these naming rules strictly. Clarity is the top priority.

- Be as clear and specific with variable and method names as possible
- **Optimize for clarity over concision.** A developer with zero context on the codebase should immediately understand what a variable or method does just from reading its name
- Use longer names when it improves clarity. Do NOT use single-character variable names
- Example: use `originalQuestionLastAnsweredDate` instead of `originalAnswered`
- When passing props or arguments to functions, keep the same names as the original variable. Do not shorten or abbreviate parameter names. If you have `currentCardData`, pass it as `currentCardData`, not `card` or `cardData`

### Code Clarity

- **Clear is better than clever.** Do not write functionality in fewer lines if it makes the code harder to understand
- Write more lines of code if additional lines improve readability and comprehension
- Make things so clear that someone with zero context would completely understand the variable names, method names, what things do, and why they exist
- When a variable or method name alone cannot fully explain something, add a comment explaining what is happening and why

### Swift/SwiftUI Conventions

- Use SwiftUI for all UI unless a feature is only supported in AppKit (e.g., `NSPanel` for floating windows)
- All UI state updates must be on `@MainActor`
- Use async/await for all asynchronous operations
- Comments should explain "why" not just "what", especially for non-obvious AppKit bridging
- AppKit `NSPanel`/`NSWindow` bridged into SwiftUI via `NSHostingView`
- All buttons must show a pointer cursor on hover
- For any interactive element, explicitly think through its hover behavior (cursor, visual feedback, and whether hover should communicate clickability)

### Do NOT

- Do not add features, refactor code, or make "improvements" beyond what was asked
- Do not add docstrings, comments, or type annotations to code you did not change
- Do not try to fix the known non-blocking warnings (Swift 6 concurrency, deprecated onChange)
- The internal project directory, target, scheme, and Swift module are all named `Clawdy` (matching the product name); keep them consistent
- Avoid running `xcodebuild` from the terminal for routine work — it can invalidate TCC permissions (use it for the build/test gate, then re-grant permissions when running the app)
- Do not re-introduce the Cloudflare Worker, API keys, or cloud STT/TTS — the app is intentionally free and local

- Commit messages: imperative mood, concise, explain the "why" not the "what"
